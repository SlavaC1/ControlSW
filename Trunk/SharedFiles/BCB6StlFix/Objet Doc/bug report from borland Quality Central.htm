Description
My program worked fine in Builder4. A recent upgrade to Builder6 with patch#4 applied showed this program. Basically my app is multi-threaded that collects data from a USB2.0 device. The app uses a STL list for each thread to hold commands/messages/data blocks. Pretty regularly the whole app will stall for 4 or 5 seconds. A pause of the app during this time always shows that two threads have called pop_front() at the same time.
This should not be a problem, like I said it worked fine with Builder4. For now I'm ditching Builder6 but would like to upgrade in the future.
Steps to Reproduce:
This is a vendor issue.

Russell Hind at 4/14/2004 1:37:28 AM -
This is due to a bug in STL Port that ships with BCB6.  To fix it you need to re-compile STLPort.

Download the public beta of STLPort from the BDN (you'll have to search for it), then edit

stl\_config.h

At line 265 you'll see

#   elif defined (_STLP_WIN32) && ! defined (_STLP_PTHREADS)

after this, add

#    define _STLP_WIN32THREADS

The bug is that without _STLP_WIN32THREADS, the allocator doesn't call sleep if the spin-lock is locked, and so high priority threads sit waiting on a spin lock the lower priority thread holds, but the lower priority thread can't run because the higher priority thread doesn't sleep.

Re-build STLPort in a separate directory (instructions are included in the download) then copy it in to your BCB6 installation (all headers and libs and dlls).  You need to re-arrange the header structure slightly as the download has a different structure to what BCB requires, but if you look in your BCB6 include directory, you will see how the directories need to be arranged.  I think the directories needed are

stl
stlport
config

stl and config I thnk are subdirectories of stlport in the download, but don't want to be subdirectories in BCB6 include directory

IMHO, this is a major bug and I have reported it to a Borland engineer last year, but nothing has been done about it.

Cheers

Russell

Russell Hind at 4/14/2004 1:41:14 AM -
FWIW, this is fixed in all later builds of STLPort, but no-one has been able to build a later version of STLPort from www.stlport.org with BCB6.  But the downloadable beta from BDN does build correctly, so you can patch it yourself to fix the problem.

Thanks

Russell

Russell Hind at 4/15/2004 1:04:20 AM -
It is resolvable.  I only discovered the problem after the last patch for BCB6 was released.  I posted a message on the newsgroups regarding it and no-one else seemed interested so I guess no one is aware of the problem.  It you look in the change log for STLPort you will see a note saying the #define was re-added which I how I found out the fix (after stepping through the allocator code to release it wasn't sleeping, because it was #defined out).

So therefore I guess they don't see it as a major problem.

This is the most public list of bugs available so is the best place for it.

Another solution is to tell BCB6 to use the RogueWave STL that shipped with BCB4 and 5, defining _USE_OLD_RW_STL to the project wide defines.  AFAICT, RW didn't have this problem.

Cheers

Russell

Andre kaufmann at 6/8/2004 7:29:04 AM -
The problem may be caused also by heap allocation / deallocation
on systems with a hyperthreading CPU.

The BCB6 memory manager uses InitializeCriticalSection to create
critical sections which are protecting the heap.
After recompiling the BCB6 libraries with InitializeCriticalSectionAndSpinCount
(...., 5000) no stalls occured anymore.
